# SSH Session And Attacker Interaction Flow

FunnyPot models an SSH honeypot session as a connection session plus one or more shell or exec channel sessions. A connection gets a short random `sessionKey`; each shell or exec channel gets a separate `sessionId` used for fake filesystem state, command queueing, and command analytics.

## Connection Handling

- `StartSshServer` binds the configured port and banner, adds an RSA host key, and registers `OnConnectionAccepted`.
- `OnConnectionAccepted` allocates a `sessionKey`, captures the remote endpoint and selected banner, enforces `MaxSessions` with a semaphore, logs `session_start`, and registers service handlers.
- On disconnect, it logs `session_end`, removes per-connection dictionaries, decrements active connection count, and releases the connection semaphore.

## Authentication Flow

- `SetupUserauth` tracks attempts by remote IP, not by source port, using `GetRemoteAttemptKey`.
- Password authentication succeeds when the attacker uses configured `SSH_USER` and `SSH_PASSWORD`, or when failed attempts reach `PASSWORD_HARVEST_ATTEMPT`.
- Non-password auth methods are logged but rejected.
- Every password attempt is logged as `auth_attempt`, and password attempts are also logged as `harvested_credential`.
- Auth telemetry includes username, password for password auth, method, key metadata, attempt number, acceptance reason, credential entropy, previous credential distance, and fingerprint hash.

## Shell And Exec Flow

- `SetupShell` accepts shell and exec channel opens.
- Interactive shells get a prompt generated by `GetPrompt`; exec commands are processed once and then closed.
- Each shell starts a `SessionCommandWorker` dedicated thread so command processing runs in order and away from FxSsh callbacks.
- The idle timer closes inactive sessions after the configured `SESSION_IDLE_TIMEOUT_SECONDS`.
- `exit`, `quit`, and `logout` close the shell with a terminal-like close message.

## Command Lifecycle

1. Attacker input is echoed and buffered until newline for interactive shells.
2. `ProcessLine` increments command counters and logs the raw command as `command`.
3. `DataHarvester.AnalyzeCommand` derives discovery, persistence, payload, egress, tunneling, persona-breakout, semantic-complexity, asset-value, and MITRE tactic fields.
4. `InputValidator` rejects empty input, overly long input, null bytes, and repetitive noise.
5. `CommandResolver.ResolveCommandAsync` selects the response path.
6. The response is sent to the channel, logged as `command_result`, and followed by a prompt for interactive shells.
7. Shell finalization logs `shell_session_end`, updates global stats, drops fake filesystem state, disposes the command worker, and triggers a Git publication attempt.

## Verified Behaviors

- Unit tests cover remote endpoint normalization, input validation, command worker ordering, built-in command behavior, static response classification, LLM routing for complex commands, and shell prompt constraints.
